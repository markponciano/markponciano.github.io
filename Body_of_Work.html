Making the software actually reliable

I set up GHL so the business can actually rely on it.

GoHighLevel stayed the friendly screen everyone uses. The actual work moved into solid code underneath, where mistakes get caught automatically.

GoHighLevel as the front face, with solid code running the show Jun 19 to Jul 8, 2026 Live

GoHighLevel stayed the customer-facing surface everyone clicks. Underneath, I built a PHP 8.2 backend layer with about 50 classes that runs the business: bookings, quoting, invoicing, payments. Booking status is derived from facts like 'quote was accepted' and 'payment cleared', not from someone remembering to move a card. Deployed as WordPress plugins.

Switched to the new system July 4 with zero downtime Executed Jul 4, 2026 Live

Every change tested end-to-end before it ships Live

358 automated tests walk through every booking scenario from capture through payment, checking that money is accounted for correctly at every step. Every code change must pass five gates before shipping: linter check, custom collision detector, code-style verification, static analysis, and the full test run.

Database changes that do not break things Live

26 numbered SQL migrations, all additive. I built a custom runner and a version ledger that tracks which migrations have been applied. The ledger caught real production drift the day it went live, meaning small silent mistakes stopped being possible.

Caught and fixed the duplicate invoice bug Jul 7, 2026 Live

Two quick Approve clicks on the same booking could produce two Xero invoices. Found the race condition, fixed it with an atomic database operation, then audited every similar transaction to ensure the same bug could not hide elsewhere.

Monitoring so problems cannot hide Jul 6 to 7, 2026 Live

Alert recorder with dedup and escalation on every scheduled task. External dead-mans-switch that pages me if the site goes silent. Live operator page showing heartbeat status and a plain-English activity feed so I can see what is happening anytime.

Finance operations and finding the money hiding in the numbers

I run the weekly numbers and find what is hiding in them

The weekly work that keeps a business grounded and honest: about 43 deliverables across 10 types, unbroken every week since February.

Found and reclaimed $8,000 AUD from GPs who got paid twice 2025, AU medical clinic Shipped

The clinic switched billing models mid-stream (old: clinic collects and pays GPs; new: GPs collect their own fees and invoice the clinic). For a few weeks both ran in parallel, so 40 GPs got paid twice. I cross-referenced the payments export against the service log to pin each person's overpayment, totalling $8,000. Then set it up as painless installments through the weekly clinic-fee invoice so no one took a sudden hit.

Found £80,000 a year hiding in the weekly P&L Apr 2026 Shipped

The weekly profit report ran on a cost configuration that had quietly drifted from what Xero actually showed. I reconciled both side by side over four months and found about £1,540 a week (£80,000 annualised) in real costs missing from the model. Profit had been overstated by the same amount. Now the weekly number tells the truth.

Rebuilt the weekly P&L from source data 6-week backfill + weekly Recurring

Rebuilds and validates the weekly profit-and-loss from source data (sales, COGS, waste, crew food, wages) against a fixed template. Catches gaps and applies config-driven overrides.

Weekly multi-currency VA payroll Weekly since February 2026 Recurring

Consolidates virtual-assistant hours per person, applies their individual rates, converts USD to GBP at the day's exchange rate, and formats it as a Monday.com board import ready to pay. The single most consistent stream in my record: unbroken every week for over five months.

Matchday sales to Xero, end to end 5+ fixtures, Apr to May Recurring

After each football matchday, takes the till sales per unit, applies rent (25%) and admin charges, produces a Monday.com import, and generates a Xero invoice with the right account codes and VAT. A complete journey from register to accounts.

Square analytics: tracking discounts, voids, and cost leaks Weekly, April Recurring

Breaks down Square discounts and voids by category against industry benchmarks, flags control gaps (like anonymous discount codes making up 30% of spend), and reviews supplier price changes for COGS-distorting unit-of-measure errors.

When a new supplier bid arrives, I compare the numbers 2026 Shipped

Compared the new supplier's line items to the current supplier's, item by item, and delivered a keep-or-switch recommendation to the store manager so the decision was made on numbers, not the sales pitch.

What this uses: Xero · Dext · Best Practice · Square · MarketMan · Monday.com · advanced Google Sheets and Excel modelling · multi-currency and multi-entity reconciliation · GST/VAT · bookkeeper's data-integrity discipline

Making the tools talk to each other

I wire separate tools together so they share one truth.

Finance, CRM, calendar, and email all pulling from the same place so they agree on what is real.

Xero, GoHighLevel, email and calendar, all wired together Live

Invoice and payment webhooks with HMAC signature verification, one-way invoice mirroring to GoHighLevel, two-way calendar sync with an echo-guard to prevent looping on the same events, and email delivery webhooks. All with idempotency handling so a retried message does not create a duplicate.

An AI assistant plugged into the WordPress site Jul 3, 2026 Working

I connected Claude to the live WordPress site's MCP server over authenticated HTTP, so it audits pages and, with approval, fixes SEO issues directly in the code. Rather than just making recommendations, it makes the actual changes.

What this uses: Xero API · GoHighLevel API · Brevo · Google Calendar API · Model Context Protocol · webhook security (HMAC) · idempotency · echo-guards · event-driven design
