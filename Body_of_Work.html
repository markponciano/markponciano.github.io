Making the software actually reliable

I set up GHL so the business can actually rely on it.

GHL stays as the friendly screen everyone uses. The actual work — bookings, invoicing, payments — moved into proper code underneath, where it can be checked and trusted.

GHL as the front face, with a proper backend running the showJun 19 – Jul 8, 2026Live

The CRM (GoHighLevel) stayed as the customer-facing surface everyone knows and clicks. Underneath, a PHP 8.2 domain layer (~50 classes) took over the automation — bookings, quoting, invoicing, payments — with a database as the single source of truth. Booking state is derived from facts (quote accepted, op-form in, invoiced, paid), never from someone remembering to move a card. Deployed as WordPress mu-plugins. This is the live nervous system of the business.

Careful cutover to the new setup, no downtimeExecuted Jul 4, 2026Live

Moved a live revenue path off its old rails with an ordered, per-step-reversible runbook and a ~130-check pre-flight gate battery. Nothing broke. Nothing paused for customers. The kind of switch-over most small businesses avoid attempting because it's usually where things go sideways.

Rigorous testing before anything shipsLive

358 automated tests including a full lifecycle regression that walks a booking end-to-end (capture, quote, accept, op-form, invoice, pay, confirmed), asserting the money at every step. Every change gated on five checks — lint, a custom collision check, PSR-12 style, PHPStan, and the full test run — before it can touch production.

Safe, reversible changes to the databaseLive

26 numbered, additive SQL migrations with a custom runner and a version ledger. The ledger caught real production drift the day it was introduced — meaning small silent mistakes stopped being possible.

Found and fixed a real bug that was creating duplicate invoicesJul 7, 2026Live

Two Approve clicks in quick succession could produce two Xero invoices for the same booking. Found the race condition, root-caused it, fixed it with an atomic DB claim, then audited and hardened every similar "once-only fact gates a side-effect" path so the same class of bug can't reappear.

Monitoring so problems can't hideJul 6–7, 2026Live

An alert recorder with dedup and P1/P2 escalation wired into every scheduled task, an external dead-man's-switch that pings me if the site goes silent, and a live operator page with a heartbeat pill and a plain-English activity feed.

What this uses: PHP 8.2, MySQL, WordPress mu-plugins · webhook and API integrations · automated testing (PHPUnit) · PHPStan and PSR-12 · GitHub Actions CI · database migrations · atomic DB operations · alerting and dead-man's-switch monitoring.

Finance operations, and finding the money in the numbers

I run the weekly numbers, and I find what's hiding in them.

The recurring, reliable work owners never see and always rely on. About 43 deliverables across 10 types, unbroken every week since February.

Reclaimed $8K AUD from double-paid GPs at a clinic2025, AU medical clinicShipped

The clinic was moving from one billing model to another (old: clinic collects and pays GPs; new: GPs collect and clinic invoices them for fees), and during the transition some weeks accidentally ran both — so GPs got paid twice. I cross-referenced the payments export against the service export to pin each GP's true overpayment, about $8K across 40 accounts. Then set up a painless installment recoupment through the weekly clinic-fee invoice, so no doctor ever had a lump-sum clawback.

Found ~£80k a year hiding in the weekly P&LApr 2026Shipped

The weekly profit-and-loss report ran on a cost configuration that had quietly drifted from what Xero actually showed. I reconciled the two side-by-side over four months. About £1,540 a week (~£80k a year annualised) in real costs were missing from the model. Profit had been overstated by the same amount. Now the weekly number tells the truth.

Weekly P&L, rebuilt from source data6-week backfill + weeklyRecurring

Rebuilds and validates the weekly profit-and-loss from source data — sales, COGS, waste, crew food, wages — against a fixed template, catching gaps and applying config-driven overrides.

Weekly multi-currency VA payrollWeekly since February 2026Recurring

Consolidates virtual-assistant hours per person, applies rates, converts USD to GBP at a captured exchange rate, formatted as a payroll-ready board import. The single most consistent stream in my record — unbroken every week for over five months.

Matchday sales to Xero, end to end5+ fixtures, Apr–MayRecurring

Per football matchday: takes per-unit till sales, applies rent (25%) and admin charges, produces a board import, and generates a Xero-ready sales invoice with account codes and VAT. A complete source-to-invoice pipeline.

POS analytics: discounts, voids, and COGS integrityWeekly, AprilRecurring

Breaks down Square discounts and voids by category against industry benchmarks, flags control gaps (e.g. anonymous discount codes making up 30% of spend), and reviews supplier price changes for COGS-distorting unit-of-measure errors.

Supplier cost analysis when a new vendor bid2026Shipped

Compared the incoming supplier's line items to the current suppliers, item by item, and delivered a keep/switch recommendation to the store manager so the decision was made on numbers, not on the sales pitch.

What this uses: Xero, Dext, Best Practice, Square, MarketMan, Monday.com · advanced Google Sheets and Excel modelling · multi-currency and multi-entity reconciliation · GST / VAT · a bookkeeper's data-integrity discipline applied at scale.

Making the tools talk to each other

I wire separate tools together so they share one truth.

The join between finance, CRM, calendar and email that most SMB stacks don't have — and that most tools alone can't do.

Xero, GHL, email and calendar, all wired togetherLive

Invoice-issue and payment webhooks with HMAC signature verification, one-way DB-to-GHL card mirroring, two-way calendar sync with an echo-guard so we don't loop on our own events, and email-delivery webhooks. All with idempotency handling so a retried message doesn't create a duplicate.

An AI assistant plugged into the WordPress siteJul 3, 2026Working

Wired Claude to the live WordPress site's built-in MCP server over authenticated HTTP, so it can audit the site and (when approved) auto-fix specific SEO issues in place, rather than just recommending changes.

What this uses: Xero API, GoHighLevel API, Brevo, Google Calendar API, Model Context Protocol (MCP) · webhook security (HMAC / shared-secret) · idempotency and echo-guards · event-driven design.
